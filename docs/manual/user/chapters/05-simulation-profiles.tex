\chapter{配置 simulation 与模型 profile}

第 3 章介绍了 \yamlkey{simulation} 块的整体结构。本章深入：内置 model profile 是什么、如何用 profile、如何不用 profile（自定义）、如何配置多 simulation 对比，以及 \yamlkey{time_alignment} 三种时间对齐策略的实战权衡。

\section{什么是 model profile}

一个陆面模型的输出 NetCDF 文件中，物理变量名是模型私有命名（CoLM2024 把 ET 叫 \texttt{f\_lh\_vap}，CLM5 叫 \texttt{QFLX\_EVAP\_TOT}，ERA5LAND 叫 \texttt{e}）。\openbench{} 用统一的标准变量名（\var{Evapotranspiration}）做评估，因此需要在两者之间建立映射。

\textbf{model profile} 就是这层映射的封装：

\begin{itemize}
  \item 模型名（如 \texttt{CoLM2024}）
  \item 数据形态（grid/stn）、默认时空分辨率
  \item 标准变量名 → NC 内变量名 + 单位 + 文件命名约定
  \item 可选的 fallback（同义变量替代）与单位换算表达式
\end{itemize}

profile 让用户的 \yamlkey{simulation} 配置极简：

\begin{minted}{yaml}
simulation:
  CoLM2024_test:
    model: CoLM2024            # 引用 profile 名
    root_dir: /data/colm2024   # 我的 NC 文件在哪
\end{minted}

只两行 + profile 自动补全其它细节。

\section{使用 profile 的最小配置}

profile 配上 \yamlkey{root_dir} 即可工作：

\begin{minted}{yaml}
simulation:
  MyRun:
    model: CoLM2024
    root_dir: /data/my_colm2024_run
\end{minted}

\openbench{} 自动从 profile 读：
\begin{itemize}
  \item \yamlkey{data_type}: \texttt{grid}
  \item \yamlkey{tim_res}: \texttt{Month}
  \item \yamlkey{grid_res}: \texttt{0.5}
  \item \yamlkey{variables}: profile 中所有变量映射
\end{itemize}

只要 \file{/data/my\_colm2024\_run/} 下的 NC 文件命名遵循 profile 约定（默认 \texttt{<varname>\_<year>\_*.nc} 或 \texttt{*.nc}，多年合并），评估即可启动。

\section{字段覆盖：profile + 局部修改}

profile 的字段都可在 simulation entry 局部覆盖：

\begin{minted}{yaml}
simulation:
  CoLM2024_HiRes:
    model: CoLM2024
    root_dir: /data/colm2024_hires
    grid_res: 0.25       # 覆盖 profile 默认 0.5
    prefix: case01_      # 文件名前缀（如果有）
    suffix: _hist        # 文件名后缀
    data_groupby: Year   # 按年分文件 vs 单文件包全部
\end{minted}

\subsection{覆盖单个变量映射}

如果你的 NC 把 \var{Latent\_Heat} 叫 \texttt{lhf}（不是 profile 默认的 \texttt{f\_lh}），可以单独覆盖那一个变量：

\begin{minted}{yaml}
simulation:
  CoLM2024_custom:
    model: CoLM2024
    root_dir: /data/run
    variables:
      Latent_Heat:
        varname: lhf       # 覆盖 NC 内变量名
        varunit: W/m2
\end{minted}

profile 中其他变量（GPP、ET 等）保持默认。这是 \emph{深合并}：profile.variables 与 entry.variables 按变量名 key 合并，而不是整体替换。

\section{自定义模型（不用 profile）}

如果你的模型不在 21 个内置之列，有两条路：

\subsection{方式 1：现场内联定义}

在 simulation entry 直接写完整变量映射，不引用任何 profile：

\begin{minted}{yaml}
simulation:
  MyCustomModel:
    model: MyCustomModel       # 名字仅用于标识；不需要在 registry 里
    root_dir: /data/mymodel
    data_type: grid
    grid_res: 1.0
    tim_res: Month
    data_groupby: Year
    variables:
      Evapotranspiration:
        varname: my_et
        varunit: kg/m2/s
        compute: value * 86400 / 2.5e6   # → mm/day
      Latent_Heat:
        varname: my_lh
        varunit: W/m2
\end{minted}

\subsection{方式 2：注册成新 profile（推荐）}

\begin{minted}{bash}
openbench model register MyCustomModel \
  --data-type grid --grid-res 1.0 --tim-res Month \
  -v "Evapotranspiration:my_et:kg/m2/s" \
  -v "Latent_Heat:my_lh:W/m2"
\end{minted}

之后可以像内置 profile 一样在 yaml 中引用：

\begin{minted}{yaml}
simulation:
  Run_v1:
    model: MyCustomModel
    root_dir: /data/v1
\end{minted}

profile 注册到用户级 catalog（\texttt{\$HOME/.config/openbench/} 或 platformdirs 默认）。后续多个 evaluation 复用。

\subsection{Fallback 与单位换算}

模型不同版本可能用不同变量名。注册带 fallback：

\begin{minted}{bash}
openbench model register CoLM2024 \
  -v "Gross_Primary_Productivity:f_gpp:g m-2 s-1" \
  -f "Gross_Primary_Productivity:f_assim:mol m-2 s-1:value * 12.011"
\end{minted}

主变量是 \texttt{f\_gpp}；找不到时回退到 \texttt{f\_assim} 并应用 \texttt{value * 12.011} 把 mol/m²/s 换算到 g C/m²/s。

\subsection{修改与移除 profile 内变量}

\begin{minted}{bash}
# 改变量映射（同 register，按名覆盖）
openbench model register CoLM2024 -v "Snow_Depth:f_snowdp:m"

# 加新变量但不动已有变量（仅追加）
openbench model register CoLM2024 --append-only \
  -v "New_Var:nc_name:unit"

# 删某变量
openbench model remove-var CoLM2024 Old_Var
\end{minted}

\section{多 simulation 对比}

\openbench{} 的核心场景之一是同一参考数据下对比多个模型。给 \yamlkey{simulation} 加多个 entry：

\begin{minted}{yaml}
simulation:
  CoLM2024_v1:
    model: CoLM2024
    root_dir: /data/colm2024_v1

  CoLM2024_v2:
    model: CoLM2024
    root_dir: /data/colm2024_v2

  CLM5_baseline:
    model: CLM5
    root_dir: /data/clm5

comparison:
  enabled: true
\end{minted}

每个 simulation 独立做评估（写到 \file{output/<run>/evaluation/<sim>/...}），然后 \yamlkey{comparison} 阶段把所有 simulation 的结果合并成对比表与对比图（portrait plot、radar 等）。

\subsection{用 \yamlkey{_defaults} 简化重复配置}

多 simulation 共享大量字段时：

\begin{minted}{yaml}
simulation:
  _defaults:
    model: CoLM2024
    data_groupby: Year
    prefix: case_

  v1:
    root_dir: /data/v1
    suffix: _hist_v1

  v2:
    root_dir: /data/v2
    suffix: _hist_v2

  v3:
    root_dir: /data/v3
    suffix: _hist_v3
\end{minted}

\subsection{标签命名建议}

simulation 标签（key 名）会进入输出目录与图例。建议：

\begin{itemize}
  \item 包含模型名 + 实验标识：\texttt{CoLM2024\_default}、\texttt{CoLM2024\_2xCO2}
  \item 不超过 ~15 字符（图例显示更清爽）
  \item 不用空格、特殊字符（用下划线分隔）
\end{itemize}

\subsection{混合 grid 与 stn sims 共用同一 reference}

\openbench{} 支持同一 reference 被 \emph{grid 类型 sim 与 stn 类型 sim 同时使用}。例如想同时检验某变量"全球网格表现"与"站点观测匹配"：

\begin{minted}{yaml}
reference:
  Streamflow: GRDC_Daily          # 站点 reference

simulation:
  CoLM2024_grid:                   # 模型在 grid 形态下的输出
    model: CoLM2024
    root_dir: /data/colm/grid

  CoLM2024_stn:                    # 同模型在站点位置抽样的输出
    model: CoLM2024
    root_dir: /data/colm/stn
    fulllist: /data/grdc_stations.csv
\end{minted}

\openbench{} 会按数据形态自动分流：grid sim → 网格评估，stn sim → 站点评估。\textbf{两类任务共享同一份 reference 预处理产物}（避免重复读 NC、重复重网格化）。

\paragraph{内部机制（避免静默故障）}

v3.0 之前混合形态在 dedupe 时只用 \yamlkey{ref\_source} 做 key，会出问题：站点预处理调用 \modname{extract_station_data_if_needed} 时\emph{会删除共享的 flat ref NC}（节省磁盘），但同一 reference 下还有 grid 任务等着读那个文件。修复后（commit \texttt{bf561bc} + 后续 backup-restore）的关键约定：

\begin{itemize}
  \item dedupe key 改为 \texttt{(ref\_source, "\_grid"|sim\_source)}：grid×grid 跨 sim 共享 prep；stn-涉及任务每对独立 prep
  \item 在 stn prep 删除 flat NC 之前，runner 先把 flat 备份到 \texttt{<flat>.unifiedmask.bak}
  \item 序列里下一个 grid 任务跑前发现 flat 缺失，从备份恢复（保留累积的 \yamlkey{unified\_mask}）
  \item 循环末尾再统一确认 flat 存在；备份用完即删
\end{itemize}

用户层面无需感知。但如果你之前在 v3.0a1 上跑过混合配置发现"评估 partial 失败但日志只有 \texttt{No such file: ...\_ref\_X\_rv.nc}"——那是已修的旧 bug，升级到 v3.0a2 即可。

\section{Time alignment 三种策略}

当多个 simulation 与多个 reference 的时间窗口不完全一致时，\yamlkey{project.time_alignment} 决定怎么处理：

\subsection{intersection（默认，推荐）}

取所有 sim × ref 的 \emph{共同} 时间窗口：

\begin{itemize}
  \item Sim A: 2000-2014
  \item Sim B: 2005-2018
  \item Ref X: 2003-2016
  \item 实际评估：max(2000, 2005, 2003) -- min(2014, 2018, 2016) = 2005-2014
\end{itemize}

优点：所有 sim×ref 比较口径完全一致，跨对可比。
缺点：浪费数据；任一 sim 时段短就拖累全局。

\subsection{per\_pair}

每对 sim×ref 各算各的重叠：

\begin{itemize}
  \item (Sim A, Ref X)：max(2000, 2003) -- min(2014, 2016) = 2003-2014
  \item (Sim B, Ref X)：max(2005, 2003) -- min(2018, 2016) = 2005-2016
\end{itemize}

优点：最大化数据利用，每对 metric 用了各自最长可用区间。
缺点：跨对 metric 不严格可比（窗口不同）；comparison 表格中要标注。

\subsection{strict}

要求 sim 与 ref 时间戳逐一精确匹配，否则报错：

\begin{itemize}
  \item 适合"完全相同年月、完全相同时间步"的对照实验
  \item 一旦窗口或频率不一致直接退出
\end{itemize}

\subsection{选择建议}

\begin{itemize}
  \item 多模型 \emph{科学比较} → \texttt{intersection}（保证可比性）
  \item 单模型评估 + 想用全部可用 reference → \texttt{per_pair}
  \item 受控对比实验（同启动、同时长） → \texttt{strict}
\end{itemize}

\section{unified\_mask：跨 sim 公平比较的第二把锁}

\yamlkey{time_alignment} 解决了"时间窗口"的可比性，但还有一个隐患：\emph{空间维度上的 NaN 分布}。每个 sim 在不同位置可能有 NaN（陆地/海洋掩码不一致、域外裁剪、模型崩格）。如果不统一，多 sim 对比就会在不同的有效格点上算 metric ——

\begin{itemize}
  \item Sim A 在 N=10000 个有效格点上算 RMSE
  \item Sim B 在 N=8500 个有效格点上算 RMSE（缺南极 + 部分海岸）
  \item 两个 RMSE 不严格可比
\end{itemize}

\yamlkey{project.unified_mask}（默认 \texttt{true}）就是这个问题的解药。

\subsection{它怎么工作}

按"累积掩码"策略：

\begin{enumerate}
  \item 第一个 sim 处理完后，遍历 ref 与 sim 的每个时空点：任一为 NaN，就把 ref 那个点也置 NaN
  \item 第二个 sim 处理时，看到的 ref 已经包含第一个 sim 的 NaN
  \item 累积下去，最终 ref 仅在 \emph{所有 sim 都有效} 的点上保留值
  \item 所有 sim 评估都跑在同一个最终 ref 上 → 同样的 N 个有效点
\end{enumerate}

实现细节（\modname{runner.local._apply_unified_mask}）：

\begin{minted}{python}
mask = np.isnan(sim_data) | np.isnan(ref_data)
ref_data[mask] = np.nan   # 写回 ref，下个 sim 见到的就是这个
\end{minted}

\subsection{何时\emph{不}应用}

\begin{itemize}
  \item \textbf{Station 评估（任一边 stn）：跳过} —— 站点数据无空间网格 NaN 概念
  \item \textbf{时间维不匹配：跳过 + WARN} —— ref/sim 时间长度或时间戳不一致时安全退出
  \item \texttt{per\_pair} 时间对齐 + grid×grid：每个 sim 用自己的 ref 副本，掩码不跨 sim 累积（避免污染）
\end{itemize}

\subsection{要不要关掉它}

\begin{itemize}
  \item \textbf{保持默认 true}：多 sim 科学对比、发表论文中的 metric 表
  \item 设 \texttt{false} 适合：
  \begin{itemize}
    \item 单 sim 评估（无累积可言）
    \item 你 \emph{故意} 想看每个 sim 在自己最大有效域上的表现
    \item 调试某个 sim 的 NaN 分布问题（不想被其他 sim 干扰）
  \end{itemize}
\end{itemize}

\subsection{陷阱}

\textbf{评估顺序敏感}：累积掩码意味着 sim 处理顺序影响中间状态。\openbench{} 内部按 sim 字典顺序处理；同一份配置每次跑顺序一致，结果可复现。但若你手动重命名 sim 标签让顺序变化，metrics 数值可能略变（虽然最终掩码是一样的，但中间 sim 看到的 ref 不同——只对带状态的 metric 有影响）。

\textbf{时间不匹配静默跳过}：sim 与 ref 时间戳不一致时 \openbench{} 只打 WARN 跳过 unified\_mask（不报错），评估继续但失去掩码统一保证。\cli{check} 不会提前抓到这种情况——只在 run 阶段日志里看到：

\begin{minted}{text}
[WARN] Unified mask: time values mismatch for ET (lengths equal but
       timestamps differ), skipping
\end{minted}

发现这条警告就要去查 ref/sim 的 time 坐标是否真的能对齐。

\subsection{累积态如何在 grid→stn→grid 序列下保住}

\openbench{} v3.0 修复了一个隐藏的 unified\_mask 数据正确性 bug：当一个 reference 同时被 grid 和 stn 形态的 sim 使用时，stn 预处理过程会删除累积了 mask 的 flat NC，导致下一个 grid sim 看不到累积态。

修复方案（\modname{runner.local} 内部 \texttt{\_backup\_flat\_ref} / \texttt{\_restore\_flat\_ref\_if\_missing}）：

\begin{enumerate}
  \item 每次 \modname{\_apply\_unified\_mask} 把累积 mask 写回 flat NC 后，runner 立即把 flat 备份到 \texttt{<flat>.unifiedmask.bak}
  \item 任何后续操作发现 flat 缺失（典型情形：stn 预处理刚删过），从 \texttt{.bak} 恢复（保留全部累积 mask）
  \item 整个变量处理完成后，备份文件自动清理
\end{enumerate}

用户层面：你不需要做任何事。\yamlkey{unified_mask: true} 在所有合法配置下都正确累积——包括混合 grid+stn 序列、\texttt{intersection} 与 \texttt{per_pair} 两种 \yamlkey{time_alignment}。这一段写在这里只是把"为什么不会出错"明确化，便于审稿/复现验证时引用。

\section{站点 simulation：fulllist 的作用}

只要 sim 或 ref 任一是 station 数据，\openbench{} 就需要一份 \emph{站点列表}（station\_list）来驱动评估。来源优先级：

\begin{enumerate}
  \item \yamlkey{simulation.<label>.fulllist}（用户显式指定）
  \item ref 数据集 catalog 中的 \yamlkey{fulllist}
  \item 上次评估生成的 \file{stn\_<ref>\_<sim>\_list.txt}
  \item 自动扫描 sim 目录（仅当 sim 是 station）
\end{enumerate}

CSV 格式必含 \texttt{site, lat, lon} 三列：

\begin{minted}{text}
site,lat,lon,IGBP
US-Ha1,42.5378,-72.1715,DBF
US-MMS,39.3232,-86.4131,DBF
DE-Tha,50.9624,13.5651,ENF
...
\end{minted}

\subsection{场景 A：grid sim × station ref（最常见）}

模型是 grid 输出，参考是站点观测：

\begin{minted}{yaml}
simulation:
  CoLM2024_PLUMBER:
    model: CoLM2024
    root_dir: /data/colm2024
    fulllist: /data/PLUMBER2/list/PLUMBER2.csv  # 必填
\end{minted}

\openbench{} 在 grid sim 输出中按 fulllist 中的经纬度做最近邻或双线性采样，得到"站点位置的模型值"，与 station ref 观测对比。

\subsection{场景 B：station sim × station ref}

两个站点 dataset 直接对比（如同时段两个 PLUMBER2 版本、或某个已采样站点的 reference 自比）：

\begin{minted}{yaml}
simulation:
  PLUMBER2_v2:
    model: PLUMBER2-EXP        # 或自定义站点 model profile
    root_dir: /data/plumber2_v2
    data_type: stn             # 显式标 station
    # fulllist 可省 —— 会从 sim NC 文件自动扫描

reference:
  Latent_Heat: FLUXNET_PLUMBER2
\end{minted}

\openbench{} 行为：

\begin{itemize}
  \item 从 sim 目录扫出全部站点（或读 \yamlkey{fulllist} 若给）
  \item 与 ref 站点列表取交集（同 site ID）
  \item 在交集站点上逐站做 metric / score
  \item 不在交集的站点 WARN 跳过
\end{itemize}

\warninline{site ID 必须能对上 —— FLUXNET 用 \texttt{US-Ha1} 这种命名约定，PLUMBER2 与 GRDC 各自有自己的 ID 体系。混合源对比前先用 \texttt{ncdump -v site} 看两边 ID 风格是否一致。}

\subsection{场景 C：station sim × grid ref（罕见）}

模型已经是站点输出，但 reference 只有 grid：

\begin{minted}{yaml}
simulation:
  PLUMBER2_sim:
    model: PLUMBER2-EXP
    root_dir: /data/plumber_runs
    data_type: stn
    fulllist: /data/plumber/sites.csv

reference:
  Latent_Heat: ERA5LAND  # grid
\end{minted}

\openbench{} 把 grid ref 在 fulllist 站点位置采样，再与 station sim 对比。逻辑与场景 A 对称。

\subsection{场景 D：grid × grid（默认，不需要 fulllist）}

无站点处理；两边重网格化后逐像素对比。本节其余内容跟你无关。

\subsection{自动扫描机制}

当 \yamlkey{sim.data_type: stn} 但 \yamlkey{fulllist} 没给时，\openbench{} 会：

\begin{enumerate}
  \item 扫 \yamlkey{simulation.<label>.root_dir}
  \item 打开第一个 NC 文件，读 \texttt{site} 坐标与对应 \texttt{lat}/\texttt{lon}
  \item 生成 \file{output/<run>/scratch/auto\_stn\_list.csv}
  \item 后续按这个 list 跑
\end{enumerate}

自动扫描适合"sim 与 ref 是同一套站点"的场景；如果 sim 站点是 ref 站点的子集或超集，建议显式给 \yamlkey{fulllist}。
